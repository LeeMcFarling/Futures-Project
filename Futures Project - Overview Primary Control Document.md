\f0\fs24 \cf0 # Futures Project \'97 Overview Primary Control Document\
\
## Purpose\
This document is the authoritative coordination and control layer for the entire Futures Project. It defines first-principles orientation, system design rules, phase discipline, and dependency constraints that govern all policy domains and subdomains.\
\
This document does **not** propose policy, design programs, or prescribe messaging. Its role is to constrain, align, and sequence work so that ambition does not exceed execution capacity.\
\
---\
\
## First-Principles Orientation\
\
The Futures Project is an institution-building effort under conditions of political, fiscal, and informational uncertainty. Policy is treated as system design rather than moral arbitration or ideological signaling.\
\
Core principles:\
- Institutional durability over rhetorical completeness  \
- Modularity and fault tolerance over elegance  \
- Separation of strategy, decision rules, and tactics  \
- Constraints (legal, fiscal, political) are binding  \
- Reversible, staged change preferred to brittle transformation  \
\
Explicit non-assumptions:\
- No expectation of consensus  \
- No reliance on perfect actors or compliance  \
- No requirement that new technology succeed for core reforms  \
\
---\
\
## Phase Model (Global)\
\
Phase 0 \'97 Problem Framing  \
Phase 1 \'97 Strategic Constraints  \
Phase 2 \'97 Structured Exploration  \
Phase 3 \'97 Outcomes & Decision Rules  \
Phase 4 \'97 Sequencing & Execution Design  \
Phase 5 \'97 External Validation  \
Phase 6 \'97 Messaging & Rebuttal\
\
No domain, subdomain, or artifact may exceed the declared phase.\
\
---\
\
## Dependency & Fault-Tolerance Rules\
\
- Domains are logically independent  \
- Failure in one domain must not cascade into others  \
- Cross-domain links must be explicit and typed  \
\
Allowed dependency types:\
- Optional integration  \
- Soft dependency  \
- Hard dependency (rare; requires fallback design)\
\
---\
\
## Messaging Separation Rule\
\
Messaging is downstream of design.  \
Design changes only through formal phase rollback.\
\
---\
\

\
Changes to this document are rare, versioned, and must explicitly surface tradeoffs.}
